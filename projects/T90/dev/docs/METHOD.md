## 1. Can the system infer likely T90 overspec?

Answer: yes, but as a probabilistic risk score rather than an exact T90 prediction.

The current implementation can estimate whether the current condition is likely to produce `T90 > 8.70`. The practical form is:

- not “the exact next T90 value”
- but “how similar the current context is to historical overspec contexts”

The current dev replay uses the local overspec ratio inside the selected stage-aware neighborhood as the primary risk score.

Interpretation:

- `AUC ≈ 0.75` means the current context retrieval is already useful for ranking “more likely to overspec” versus “less likely to overspec”.

## 2. How is recommendation accuracy measured under a given condition?

The current recommendation is evaluated against the true LIMS calcium/bromine values in offline replay.

There are two main accuracy views:

1. Point accuracy
   - compare the recommended `best_point.calcium` and `best_point.bromine` against the actual lab calcium/bromine
   - metric: `MAE` and `P90 absolute error`

2. Range usefulness
   - check whether the actual lab calcium/bromine falls inside the recommended calcium/bromine interval
   - metric: `range coverage ratio`

Recommended interpretation:

- `MAE` tells us whether the recommended point is numerically close to the real historical operating result
- `P90` tells us whether the recommendation is stable in the worse-but-common tail
- `coverage ratio` tells us whether the recommended interval is practically usable, not only whether the single best point is close

## 3. Current implementation method

### 3.1 Target and deployment role

The current implementation is not built around “predict exact T90 from DCS”.

It is built around:

- identify the current process state from recent online data
- retrieve similar historical cases
- recommend calcium/bromine values or value ranges that historically fit the target T90 window

Current target window:

- `8.45 ± 0.25`
- equivalent to `8.20 ~ 8.70`

### 3.2 Data flow

Offline data sources:

- DCS: `merge_data.csv`, `merge_data_otr.csv`
- LIMS: `t90-溴丁橡胶.xlsx`
- PH: `B4-AI-C53001A.PV.F_CV.xlsx`

Offline asset-building flow:

1. Read DCS, LIMS, and PH.
2. Group LIMS rows by `sample_time`, because LIMS is not one-row-per-sample.
3. For each LIMS sample time, extract the latest `50min` DCS window before that sample.
4. Encode each DCS window into numerical summary features.
5. Build a base `50min DCS` casebase.
6. Build a `PH(120min)`-augmented casebase for the stage that benefits from PH.
7. Fit a DCS-only stage identifier and export the stage policy JSON.

Runtime online flow:

1. Input current `50min` DCS window.
2. Encode the DCS window into summary features.
3. Use the stored stage identifier to decide which stage the current context belongs to.
4. Read the stage policy:
   - keep pure DCS recommendation, or
   - if allowed, add `PH(120min)` enhancement
5. Retrieve similar historical cases from the chosen casebase.
6. Fit a local calcium/bromine feasibility model.
7. Output:
   - recommended calcium range
   - recommended bromine range
   - recommended best point
   - stage decision
   - whether PH was actually used

### 3.3 Intermediate models and components

#### A. DCS window encoder

Input:

- recent `50min` DCS window

Per selected sensor, encode:

- `last`
- `mean`
- `std`
- `min`
- `max`
- `delta`
- `valid_ratio`
- `slope`

This converts a raw time window into a fixed-length numeric state vector.

#### B. Stage identifier

Used only on DCS-encoded features, not on lab-only values.

Pipeline:

- `SimpleImputer(strategy="median")`
- `StandardScaler()`
- `KMeans(n_clusters=3)`

Important:

- the deployed stage identifier is `DCS-only`
- it does not require current calcium, bromine, or current T90

#### C. PH enhancement

PH is optional, not mandatory.

Current second-phase policy:

- `stage_0`: PH disabled
- `stage_1`: PH disabled
- `stage_2`: PH enabled with `120min` lag

At runtime, when PH is enabled:

- compute lag-aligned PH features at `runtime_time - 120min`
- use a `50min` PH feature window

PH features are:

- `ph_point`
- `ph_mean`
- `ph_std`
- `ph_min`
- `ph_max`
- `ph_delta`

If PH is not provided or cannot be aligned:

- the system falls back automatically to the stage-specific pure DCS path

#### D. Similar-case retrieval

Once the stage is known, the system works inside the corresponding historical casebase.

Pipeline:

- choose overlapping numeric context features
- use `NearestNeighbors` in encoded context space
- retrieve a local neighborhood

This neighborhood serves two purposes:

- support overspec risk estimation
- support calcium/bromine recommendation

#### E. Local calcium/bromine recommender

Inside the local neighborhood:

- fit a local `LogisticRegression` on `[calcium, bromine] -> is_in_spec`
- scan a calcium/bromine grid
- keep the feasible region above the chosen in-spec probability threshold
- output:
  - recommended calcium range
  - recommended bromine range
  - recommended best point

This is why the current system is better described as:

- a stage-aware local recommendation system
- not a global exact T90 regression model

### 3.4 Why the current method is reasonable

The current implementation is aligned with the real plant constraints:

- current calcium and bromine are not available online
- T90 is also not available online
- only DCS context and optionally PH trend are available online

So the runtime question becomes:

- “Given the current DCS context, and optionally the PH trend, which historical calcium/bromine region is more likely to satisfy T90?”

That is exactly what the current system answers.

## Current bottom-line conclusion

Question 1:

- yes, the current implementation can infer whether T90 is likely to exceed the upper spec
- this should be used as a probabilistic risk score, not as an exact value prediction

Question 2:

- recommendation accuracy should be measured by both point error and interval usefulness
- the current core metrics are calcium/bromine MAE, P90, and range coverage
- the most meaningful splits are by `stage` and by `T90 status`

Question 3:

- the current implementation is a stage-aware, case-based, locally modeled recommender
- main online input: `50min DCS`
- optional online enhancement: `PH(120min)`
- main intermediate models:
  - DCS statistical encoder
  - DCS-only KMeans stage identifier
  - nearest-neighbor case retrieval
  - local logistic calcium/bromine feasibility model

If the next step is needed, the most useful follow-up experiment would be:

- compare the current overspec risk score against a dedicated binary overspec classifier built on the same `50min DCS` stage-aware context
